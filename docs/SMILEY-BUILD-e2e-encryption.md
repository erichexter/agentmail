## Executive summary

AgentMail today ships plaintext over a cleartext LAN with a single shared bearer token. This brief specifies the E2E-encryption slice that layers **confidentiality, sender-authentication, unforgeable delivery signalling, and (incrementally) a real PKI trust root** onto that relay without breaking legacy plaintext peers. The design is the accepted PRD's **X3DH-per-message forward secrecy with NO Double Ratchet** (FLAG-1 â€” the single most important escalation, already resolved in the PRD): one-shot, unidirectional, store-and-forward delivery derives a fresh `SK` per message from an ephemeral key plus recipient prekeys, with no advancing chain state.

All cryptographic authority lives in TLV pre-images gated by committed cross-implementation test vectors, never JSON/protobuf bytes.

The brief has survived **three rounds of hostile review** (60 findings total; every valid one fixed and audited in Â§Â§12â€“14). **FLAG-40 (M-NEW-2 â€” pin the truncated-key-id length `N`, and NEVER feed a key-id into the transcript):** two defects, both fixed:
1. **`N` is pinned to the FULL SHA-256 digest: `*_key_id = base32(sha256(pubkey))` with N = 52 base32 chars (256 bits, no truncation).** Collision argument: full 256-bit digest gives ~128-bit collision resistance and eliminates the lookup-ambiguity/grinding surface a short `N` would open. (If a shorter display form is ever wanted it is a *display-only* concern and MUST NOT change the on-wire/AD-signed value.) State `N=52` in the vector header.
2. **The transcript (S4/PR2.3) consumes RAW 32-byte public-key bytes, NOT the `*_key_id` fingerprint string.** These are two different values that a confusingly-similar name invites an implementer to swap, silently desyncing `SK`. The transcript field names are explicitly documented as **"raw public key bytes (32), NOT `sender_key_id`/`recipient_key_id`."** Pin a vector that **fails** if the truncated/base32 fingerprint is fed into the transcript in place of the raw pubkey.

**FLAG-41 (M-NEW-3 â€” padding is self-describing inside the sealed plaintext; recipient recovers true length):** `pad_size_buckets` pads "the body inside the AEAD," but nothing said how the recipient recovers the true plaintext length â€” zero-padding a binary body whose real trailing bytes are `0x00` is unrecoverable. **Resolution: padding is self-describing INSIDE the AEAD plaintext.** Sealed layout is `plaintext_len(be32) â€– plaintext â€– pad_zeros_to_bucket`. The recipient reads the `be32` length prefix (integrity-protected by the AEAD tag) and slices exactly that many bytes; padding bytes are discarded regardless of value. (A PKCS#7-style trailer is an acceptable equivalent, but the explicit inner length prefix is specified as canonical.) `size`/`content_hash` cover the **full post-padding** sealed-then-ciphertext bytes (FLAG-23). **Pin a vector with a binary body whose real trailing bytes are legitimately `0x00`, asserting exact plaintext recovery.**

**FLAG-45 (S-NEW-1 â€” `reply_to` redirect is ENFORCED recipient-side):** `reply_to` is "sender-owned `name@host` only," but with no recipient check it is a reply-redirection primitive. **Rule: the recipient rejects an envelope whose `reply_to.host != from.host` (or whose `reply_to.name@host` is not the authenticated sender identity) with `reply_to_redirect`.** `reply_to` is signed, so this is a clean structural check. **FLAG-25 (R2-B2 â€” `fanout` = N independent seals under one logical `msg_id`; the seal-cache key is `(msg_id, to)`):** AD binds both `to{name,host}` and `recipient_key_id`, so a fan-out message needs **N distinct seals**. Resolution:

Combined with FLAG-31's bounded delivery window: **hold a message past `MAX_MESSAGE_LIFETIME` and it is guaranteed silently expiry-dropped at the recipient (PR1.6.2), with no receipt ever returned.** **Disclose: without transport TLS or Tailscale, an active LAN adversary has unbounded selective denial-of-delivery, and the bounded window turns any held message into permanent silent loss.** Recommend **TLS-or-Tailscale as the AVAILABILITY floor**, not merely metadata defense-in-depth. (This is distinct from the metadata/confidentiality framing â€” it is an integrity-of-*delivery* consequence and belongs next to the "frequently never-answered" framing, which otherwise masks the loss.)

**Metadata + relay-auth exposure â€” document EXACTLY and COMPLETELY what leaks in cleartext (H1, expanded per R2-H1):** the **entire outer envelope is cleartext JSON.** The COMPLETE leaked set an on-LAN observer sees without transport TLS is:
- **Routing/timing:** `from`, `to`, `msg_id`, `size` (exact, unpadded unless `pad_size_buckets`), `created_at`, `expires_at`, `protocol_version`, `content_type`, `priority`, `reply_to`.
- **All four signed booleans:** `want_receipt`, `require_e2e`, `ordered`, `fanout`.
- **MOST DAMAGING â€” persistent identity fingerprints (`enc.sender_key_id`, `enc.recipient_key_id`, `auth.key_id`):** **stable, long-term, pseudonymous identifiers that LINK every message an agent ever sends â€” even across `from`/`host` renames.** Highest-value metadata leak; call out explicitly.
- **Receipt type is observable per-message** (`consumed` vs `undecryptable` vs `bounced`) â†’ leaks decrypt success/failure to a passive observer.
- **Subject family** â€” sealed inside the AEAD (FLAG-26), so **NOT** in cleartext for enc envelopes; visible only for legacy `*.msg.md`.
- **Disposition:** the identity fingerprints and booleans are **accepted-exposed and disclosed** (sealing them would break relay routing/dedup and recipient key-selection). **Present as "complete enumeration, key-id linkability accepted-exposed, mitigated only by transport TLS + Tailscale" â€” NOT "bounded."** `pad_size_buckets` (default off) is the only knob that reduces `size`-fingerprinting.
- **Relay bearer token is sniffable on the LAN in cleartext mode.** Mitigation (PR1.12/H2): directory-mutating ops + OPK claims gate on the agent-signed record / signed fetch, so a leaked token buys **routing/DoS nuisance only** â€” not key substitution, not OPK theft. **Disclose:** in cleartext mode the relay trusts the **entire L2 segment for relay-auth**; the token is not a confidentiality boundary.
- **Availability (H-NEW-5):** without TLS/Tailscale an **active** adversary has unbounded selective drop/delay â†’ held-past-`MAX_MESSAGE_LIFETIME` messages are permanently, silently lost.
- **Not exposed:** the message **body** (E2E-sealed) and the **subject family** on enc envelopes; sender identity is cryptographically authenticated. `transport_tls` is purely defense-in-depth for the metadata above; **Tailscale-the-network is a real dependency (FLAG-10/M-NEW-1).**

## 5. Key storage on a shared-user Linux box (+ honest disclosure) â€” hardened

Layout under `~/.claude/agentmail/`: `keys/<name>@<host>/{ident.key, spk.key, opk/, retention/}`, `pki/{root.pub, issuer.cert, pinned/}`, `state/{done, epochs, watermarks, consumed_opk, quarantine/}`.
- POSIX perms: key dirs `0700`, key files `0600`, owned by the agent's user; enforce at create (`File.SetUnixFileMode`) and **assert at startup** (refuse world/group-readable private â†’ alert). **Perms buy nothing against the same-uid-sibling threat â€” siblings share the uid.**
- **In-memory protection (M6) with degraded-flag surfacing (FLAG-35):** hold `ident`/`spk`/`opk` privates in libsodium secure memory where the binding exposes it; **disable core dumps**; keep privates **out of swap** (`mlock` via libsodium secure heap); zero buffers after use.
  - **FLAG-35 (R2-M2 â€” mlock/secure-alloc failure MUST be surfaced):** an unprivileged agent often hits `RLIMIT_MEMLOCK` (commonly 64 KiB), at which point `mlock`/`sodium_malloc` **silently degrades to normal swappable heap.** **On failure, set a `swap_exposed` degraded flag, surface it in `/agents`, and alert.** Do not pretend the no-swap guarantee holds.
- **At-rest disclosure (M6 + B-NEW-4 â€” the live keys are the load-bearing fact):** `ident.key`/`spk.key`/`opk/` are stored **UNENCRYPTED** (no passphrase/OS-keyring wrapping) â€” **disk theft = full identity + recomputation of `SK` for every message currently in the inbox** (`opk`-backed or not), because a thief with the live privates + the cleartext public envelope fields (`eph_pub`, epochs) reconstructs the DHs. **This is why "FS from message 1 against disk theft" is FALSE and must never be stated â€” FS holds only for *consumed/retired* keys against a non-disk-holding adversary (FLAG-1/FLAG-34).**
  - **The `retention/` ring IS encrypted at rest (FLAG-34, PR2) under a wrapping key bound to the OS keyring/TPM (externally protected, survives restart, crypto-erasable)** â€” but this protects **superseded/retired keys only**, not in-flight messages (whose consuming keys are the still-cleartext live privates above). **Disclose file-deletion â‰  erasure** (SSD/CoW): destruction relies on crypto-erasing the wrapping key.
  - *Remediation to consider (flag to Hex, not silently omit):* keyring/TPM/passphrase-wrap `ident.key`/`spk.key`/`opk/` too â€” the only path that would extend disk-theft resistance to in-flight messages.
- **Same-user disclosure (carry forward verbatim):** on single-user boxes all agents share one OS trust domain â€” **any agent can read a sibling's `ident_priv` and pass its possession-proof; per-agent unforgeability is NOT met intra-box.** Detect at startup â†’ `isolation=same_user_degraded` (surface in `/agents`), logged + alerted. Keys buy **cross-box** unforgeability only until **container-per-agent** (preferred) or per-OS-user isolation lands. **Note this is the same shared-trust-domain collapse that makes multi-agent-per-hub `require_e2e` first-pinning unsound at the network layer (FLAG-11.2) â€” the two are the same defect at different layers.**

## 9. Residual risks Hex must accept (sign-off list)

1. **Intra-box unforgeability is NOT met (same-user trust-domain collapse).** On a single-user Linux box, every agent runs as the same uid and can read a sibling's `ident.key` and pass its possession-proof.
